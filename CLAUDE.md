# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this project is

**Atlas Project Insights** — a Next.js hackathon demo (Eurazeo AI Hackathon, Google Paris, July 9-10 2026) that ingests K-12 recruiting workflow transcripts, runs two Gemini calls (extraction → dependency graph, then graph → insight), and renders both in a two-pane UI. The hero reveal: two initiatives secretly share one person (Maria Lopez) and one legacy spreadsheet — a $34K/year single point of failure.

The full spec is in `Atlas-Project-Insights-Build-Spec.md`. Build in phases (0-7) as defined there. Stop and confirm acceptance criteria after each phase before advancing.

```bash
npm run dev      # start dev server at localhost:3000

```

## Architecture

**Data flow:** seed transcripts → `GET /api/gemini/extract` (Gemini extraction) → `Graph` JSON → React Flow map. Then: `Graph` → `POST /api/gemini/analyze` → `Analysis` (`{ processSummary, insights }`) JSON → insight panel (rendered trust → insight → action), whose `entitiesInvolved` ids are passed back to the map to highlight the involved nodes/edges. A live interview path (`components/InterviewCapture.tsx`) captures audio in the browser (MediaRecorder), transcribes each answer via `POST /api/gemini/transcribe`, builds a `Session`, and persists it via `POST /api/sessions`. From the Sessions view, "Generate process map" reconstructs each chosen session's transcript (with a `SPEAKER IDENTITY` header) and `POST`s `{ transcripts: string[] }` to `/api/gemini/extract`, feeding the same extract → analyze pipeline.

**Server / client boundary:** `app/page.tsx` is a server component — it reads `seed/graph.json` with `fs.readFileSync` and passes the parsed `Graph` (plus seed `Session[]` via `loadSeedSessions`) to client components as props. All Gemini calls happen in API routes only (`app/api/gemini/`); the key never reaches the browser. Browser audio is sent to `/api/gemini/transcribe` as raw bytes — the key stays server-side. No `NEXT_PUBLIC_` env vars exist or should be added for secrets. The shared Gemini client + helpers live in `lib/gemini.ts` (`generateWithFallback`, `transcribeAudio`, `stripFences`, `isValidSessionId`) and are imported by every route. **Do not import `lib/sessions.ts` or `lib/sessionStore.ts` into client components — they import `fs` and will break the client bundle; session status is computed server-side.**

**Transcription route** (`app/api/gemini/transcribe/route.ts`): `POST` reads the raw audio body, base64-encodes it, and calls `transcribeAudio(base64, mimeType)` (Gemini speech-to-text with model fallback on 503/429), returning `{ text }`.

**Sessions store + route** (`lib/sessionStore.ts`, `app/api/sessions/route.ts`): `saveSession`/`listSavedSessions` persist one JSON file per session under `seed/saved-sessions/` (the hackathon-local stand-in for Firestore). `GET /api/sessions` lists saved `Session[]` newest-first (pass `?projectId=…` to scope to one workspace; otherwise all saved sessions are returned and the client buckets them by `projectId`); `POST` validates the id (`isValidSessionId`), recomputes status server-side (`computeSessionStatus` in `lib/sessions.ts`), persists `projectId`/`parentSessionId` when present, saves, and returns `{ ok, session }`. `DELETE /api/sessions?id=…` removes one live-captured session file (`deleteSession`), returning `{ ok, deleted }`; it rejects invalid ids and any seed id (prefix `seed-`) with 400, since seeded demo sessions live in transcript files, not the saved store, and must never be deletable.

**App shell** (`components/AtlasApp.tsx`): client component holding a **four-view** sidebar (`NAV_ITEMS`) — **Processes** (the project overview / home, the default landing), **Sessions** (two-pane browser over the active project's `activeSessions`; saved sessions are fetched from `GET /api/sessions` on mount and re-fetched on return from an interview, and rendered ahead of the seeds), **Live interview** (`InterviewCapture` voice stage), and **Map & analysis** (`DependencyMap` + `InsightPanel`). It owns the `insights` state and derives `highlightedIds = insights.flatMap(i => i.entitiesInvolved)` to drive map highlighting. Multi-select in the Sessions view feeds the chosen sessions to `onGenerateMap`. `liveMode` defaults **off** (demo-safe). `onGenerateMap` and `runAnalysis` both gate on `liveMode`/`extractedGraph`: with `liveMode` off they reveal the prepared `graph`/analysis with **no network call**; with it on they hit `/api/gemini/extract` then `/api/gemini/analyze`. The flow is four presenter-paced beats (open session → Generate map → Run analysis → human-in-the-loop flag).

  **Project state in the shell:** `AtlasApp` composes the active workspace from `projects = [k12Project, ...userProjects]`. `k12Project` is a `useMemo` built from the `graph` + `preparedAnalysis` (`{ preparedProcessSummary, preparedInsights }`) props (never persisted); `userProjects` are fetched from `GET /api/projects` (`refreshProjects`, try/caught so the prepared workspace stands alone if the call fails). `activeProjectId` (default `K12_PROJECT_ID`) selects `activeProject` and `activeSessions = sessionsForProject(...)`. The **Processes** view renders an `overviews` array of `ProjectOverview` cards (session count, derived status badge, last-updated, Map/Analysis check dots); a top-bar **`ProjectSwitcher`** dropdown switches workspaces and a create-project modal (`name` / `lineOfBusiness` / `description`) calls `onCreate`. Per-project **reveal** state is tracked in `revealedRef` so each project's revealed map/analysis is restored (`hydrateDisplay`) on switch without re-running the beats; `selectProject` clears timers and re-hydrates. `persistProject` POSTs user projects to `/api/projects` (best-effort; never the K-12 workspace). Analysis-time `entityAttributes` are merged into `mapGraph` (the graph handed to the map) via `useMemo` for node detail cards.

  **Delete a process:** two entry points request deletion — a hover-revealed trash control on each **Processes** overview card, and a danger **"Delete this process"** item at the bottom of the header `ProjectSwitcher`. Both set `deleteTarget`, which renders a `ConfirmDeleteModal` ("Delete [name]? This removes its sessions, map, and analysis. This cannot be undone." · Cancel / danger Delete). Confirm calls `deleteProject(id)`: it optimistically drops the project and its scoped sessions from state, clears `revealedRef`, fires `DELETE /api/projects?id=…` (best-effort), and — if the deleted project was active — switches to another remaining project (else K-12) and lands on the Processes overview. Protected K-12 is guarded via `isProtectedProject`: its delete control is shown **disabled with a "Demo project cannot be deleted" tooltip** in both entry points, and the API rejects it regardless, so the hero demo can never be deleted.

  **Delete a session:** within the **Sessions** view, two entry points request deletion of a single captured session — a hover-revealed trash control on each session card, and a danger **"Delete session"** button at the bottom of the open transcript detail panel. Both call `onRequestDeleteSession`, setting `sessionDeleteTarget`, which renders the shared `ConfirmDeleteModal` ("Delete the session with [participant name]? Its transcript will be removed. This cannot be undone." · Cancel / danger Delete). Confirm calls `deleteSession(id)`: it optimistically drops the session from `savedSessions` (which automatically updates the Sessions list, the "Captured · N" count, the Generate-map "from N sessions" label, and deselects it from map generation — `selected`/`targetsForMap` derive from the `sessions` prop), fires `DELETE /api/sessions?id=…` (best-effort), and **never silently regenerates the map**. If a map is currently shown, it sets `mapStale` so the Dependency Map header shows a **"MAY BE OUT OF DATE"** badge until the presenter regenerates (cleared at the next `onGenerateMap`). Seed sessions (id prefix `seed-`) are protected: their trash/detail controls are **disabled with a "Demo session cannot be deleted" tooltip**, and `deleteSession` plus the API both reject the seed prefix, so the three demo SMEs (James, Priya, Maria) are never deletable. `ConfirmDeleteModal` is generalized to `{ title, body, onCancel, onConfirm }`, shared by both the project- and session-delete flows.

**Key types** (`lib/types.ts`): `Entity`, `Relationship`, `Graph`, `Insight`, `Analysis` (`{ processSummary, insights, entityAttributes? }`). These are the canonical shapes for everything Gemini produces and everything the UI consumes. `Session` carries an optional `projectId` (absent on legacy/seed sessions, which belong to the default K-12 workspace). `Project` is a named process-intelligence **workspace** (`{ id, name, lineOfBusiness, description?, createdAt, sessionIds, graph?, analysis?, status }`) — distinct from a `process`-type graph `Entity`; `ProjectStatus` is its pipeline lifecycle `empty → captured → mapped → analyzed`.

**Projects / multi-workspace** (`lib/projects.ts`, `lib/projectStore.ts`, `app/api/projects/route.ts`): the app is multi-workspace. Each `Project` scopes its own sessions, generated graph, and analysis. The prepared **K-12 workspace** (`K12_PROJECT_ID = "proj-k12-recruiting"`) is the demo-safe default — it is **never persisted**; it is rebuilt every load from the seed graph + prepared analysis + seed sessions, so the hero demo stays intact. User-created projects persist as one JSON file each under `seed/projects/` (`saveProject`/`listSavedProjects`, the same Firestore stand-in as sessions). `deriveProjectStatus()` maps pipeline progress to status (analysis ⇒ analyzed, graph ⇒ mapped, ≥1 session ⇒ captured). `sessionsForProject(id, saved, seed)` returns a project's sessions: saved sessions match on `projectId` (defaulting to K-12 when absent), and the K-12 workspace additionally folds in the seed sessions. `isProtectedProject(id)` guards the seeded K-12 workspace from deletion everywhere a delete is offered. `GET /api/projects` lists user projects newest-first; `POST` upserts one (rejecting the reserved K-12 id and invalid/blank ids); `DELETE /api/projects?id=…` removes a user project file (`deleteProject`) **and** every saved session scoped to it (`deleteSessionsForProject` in `lib/sessionStore.ts`), returning `{ ok, removedSessionIds }` — it rejects protected/invalid ids with 400. A project's graph/analysis live inside its JSON file, so deleting the file removes them too.

**Extraction route** (`app/api/gemini/extract/route.ts`):
- `GET` reads all `seed/transcript-*.md` files (located directly in `seed/`, not `seed/transcripts/`) and runs extraction
- `POST` accepts `{ transcripts: string[] }` in the body; persists to `seed/latest-graph.json` and, when a valid `sessionId` is given, to `seed/sessions/`
- Each transcript from the UI carries a `SPEAKER IDENTITY` header (name, role, team); `EXTRACTION_SYSTEM` rule 6 turns that into a person entity with `owns`/`hands_off_to` edges, so every SME appears in the graph with their role
- Falls back through the `GEMINI_MODELS` list in `lib/gemini.ts` via `generateWithFallback` on retryable errors — **503** (model overloaded) and **429** (quota/rate-limit exceeded); free-tier quota is per-model-per-day, so advancing to the next model recovers once a model's daily cap is hit (`shouldFallback`)
- Strips markdown fences from Gemini output before JSON parsing (`stripFences`)
- Validates that every relationship source/target references a real entity id (returns 422 on failure)

**Analysis route** (`app/api/gemini/analyze/route.ts`): returns an `Analysis` = `{ processSummary, insights: [hero, secondary] }`. The hero finding is **not left to the model**. `findCrossovers()` deterministically walks the graph, finds every entity touching ≥2 initiatives, and ranks processes first — so Eligibility Verification always wins. That detected crossover (with exact `entitiesInvolved` ids) is injected into the prompt as a hard constraint, and `hero.financialImpact` is overwritten in code with `computeFinancialImpact()` after the call. `describeFlows()` enumerates each initiative's process chain in the prompt so the model's `processSummary` reliably covers **both** flows. A secondary `needsHumanReview` insight is appended from `seed/secondary-insight-review.json`. Persists the whole `Analysis` to `seed/latest-analysis.json` (and `seed/sessions/<id>-analysis.json` when a sessionId is given). Verified 5/5 runs via `lib/phase6Check.ts`: processSummary names both flows; crossover surfaces Eligibility Verification (both initiatives, Maria, spreadsheet, $34,560) every time.

**Finance** (`lib/finance.ts`): `FINANCE_INPUTS` (editable constants — apps/week 120, min/app 8, $45/hr, 48 wks) and `computeFinancialImpact()` → the committed **$34,560/year** figure, headline, unit, and basis. Swap the constants to plug in real FP&A numbers without touching logic.

**Prompts** (`lib/prompts.ts`): `EXTRACTION_SYSTEM`, `ANALYSIS_SYSTEM`, and `INTERVIEW_SYSTEM` are the verbatim system instructions passed to Gemini. Edit these to tune behavior — they are the primary lever for output quality. `EXTRACTION_SYSTEM` rule 6 is what binds each transcript's `SPEAKER IDENTITY` header to a person entity (role + team) with ownership/handoff edges. `ANALYSIS_SYSTEM` returns one object `{ processSummary, insight }` — `processSummary` must name both initiatives' flows. The transcription instruction lives inline in `transcribeAudio` (`lib/gemini.ts`).

**Dependency map** (`components/DependencyMap.tsx`): client component, receives `graph: Graph` + optional `highlightedIds`, runs dagre LR auto-layout, renders with `@xyflow/react`. Node colors (CSS vars in `app/globals.css`): purple = initiative (`--c-init`), teal = process (`--c-proc`), amber = person (`--c-person`), pink = system (`--c-system`). Highlighted nodes pulse (`atlas-node-highlight`) and their connecting edges animate (`atlas-risk-edge`); non-highlighted nodes dim. **Clicking a node** opens a `NodeDetailCard` (top-right overlay; pane click closes it) whose contents come from `describeEntity(graph, id)` in `lib/entityDetail.ts` — a pure, deterministic derivation over graph topology. Per type: **person** → role, team, processes owned, handoff count, a SPOF risk score + level + recommended action; **process** → initiatives that depend on it, owners, systems, `hoursPerWeek` (if present), SPOF flag; **system** → processes using it, maintainers (owners of those processes), SPOF flag; **initiative** → its processes and the other initiatives it shares them with. A process is a single point of failure when ≥2 initiatives converge on it under ≤1 owner, or its sole owner runs it on a legacy system.

**Entity detail** (`lib/entityDetail.ts`): `describeEntity` returns a discriminated union (`PersonDetail | ProcessDetail | SystemDetail | InitiativeDetail`). Relationship semantics it relies on: `part_of` (process→initiative), `depends_on` (initiative→process), `owns` (person→process/initiative), `uses` (process→system), `hands_off_to` (process→process). Verified by `lib/phase10Check.ts`: Maria's card shows she owns Eligibility Verification, has 4 handoffs, and a High risk score (3); the eligibility card shows both initiatives depend on it, 16 hrs/week, SPOF.

**hoursPerWeek enrichment:** the prepared `seed/graph.json` bakes `attributes.hoursPerWeek: "16"` on the eligibility step (= 120 apps × 8 min, the finance basis) so the offline demo card shows it. On the live path, `ANALYSIS_SYSTEM` returns a top-level `entityAttributes` map (`{ procId: { hoursPerWeek } }`); the analyze route deterministically forces the shared crossover step's hours and layers the model's estimates on top, returns them on `Analysis.entityAttributes`, and `AtlasApp` merges them into `displayGraph` (passed to the map) via `useMemo`.

**Insight panel** (`components/InsightPanel.tsx`): renders in the H1 "trust → insight → action" order — (1) `ProcessSummaryCard` ("How this process works", the `processSummary`), (2) `HeroCard` (the crossover: title, explanation, confidence meter, recommended action — no money), (3) `FinancialCard` (the animated `$` count-up + basis). The `needsHumanReview` secondary follows as a `ReviewCard` with `whatToCheck` and confirm/dismiss controls. Takes a `processSummary` prop alongside `insights`.

## Out of scope (do not build)

Auth, real databases (local JSON stores under `seed/saved-sessions/` and `seed/projects/` stand in for Firestore), multi-company support, heat maps, automation discovery, any insight path beyond the single hero finding.
